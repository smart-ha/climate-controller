# climate-controller

Home Assistant интеграция, которая объединяет несколько cooling/heating устройств под одной climate-сущностью с тремя preset'ами (`work`, `chill`, `sleep`), PID-регулятором и опциональным расписанием температуры по времени суток для каждого preset'а.

Подробное описание установки и настройки — в `README.md` в корне репозитория. Этот файл фокусируется на нюансах поведения, которые легко упустить.

## Флаг `passive`

В OptionsFlow на экране **Devices & sensors** для каждого устройства из `cooling_devices` и `heating_devices` есть пара чекбоксов: **Enable** и **Passive**. Поведение определяется флагом `passive` (по умолчанию выключен):

- **`passive = false` (обычный режим).** Контроллер сам управляет устройством. В режиме `AUTO` посылает `set_temperature` (для `climate.*`) или `turn_on`/`turn_off` (для `switch`/`input_boolean`) по PID-выводу. При переходе в `OFF` контроллер физически **выключает** устройство — `switch`/`input_boolean` через `turn_off`, `climate.*` через `set_hvac_mode=off`.
- **`passive = true`.** Устройство только наблюдается. Никаких сервис-вызовов от плагина к нему не идёт — ни в `AUTO`, ни в `OFF`. Это сохраняет инвариант «не трогать»: ты сам контролируешь устройство руками или внешней автоматизацией.

Когда полезно ставить `passive`:

- Обогреватель, который ты вручную включаешь при гостях, и не хочешь, чтобы плагин его выключил при переходе в `OFF`.
- Сплит-кондиционер с собственной внутренней регуляцией — тебе нужно только видеть его state, не управлять им.
- Любая ситуация, где устройство временно сконфигурировано в плагине, но текущим режимом ты не хочешь, чтобы плагин на него влиял.

Снятие `passive` сразу возвращает устройство под управление контроллера на следующем тике `_evaluate`.
